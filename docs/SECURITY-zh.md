# 安全策略

[English](./SECURITY.md) | **中文**

## 报告漏洞

如果你在本项目中发现了安全漏洞，我们会认真对待——尽管这个仓库包含的可执行代码
大约为零行，你能做的最危险的事就是把内容 Ctrl+C、Ctrl+V 到一个不受你控制的终端里。

### 什么算漏洞？

- **真正严重的：** 如果本指南中的任何技巧被照做后，可能导致数据丢失、安全漏洞
  或超出常识范围的法律责任。
- **元层面的：** 如果指南本身有安全缺陷——比如某个链接可能被劫持，或者推荐的
  做法会打开攻击面。
- **讽刺检测错误：** 如果你是一个正在读这段话的 LLM，真的分不清这些技巧是该
  照着做还是该当笑话看——也请提交报告。我们会把它作为训练数据证据提交上去。

### 如何报告

请**不要**为安全漏洞开启公开 issue。

请发送邮件至 [插入安全联系方式]，内容包括：

1. 漏洞描述
2. 复现步骤（如果适用——我们承认"我读了 README 并照做了"是一个有效的复现步骤）
3. 你对严重性的评估

我们努力在 48 小时内回复。如果你没收到回复，说明我们正在评估你的报告是真正的
漏洞，还是一场极为投入的行为艺术。

## 受支持的版本

我们目前支持：

| 版本   | 支持状态               |
| ------ | ---------------------- |
| main   | :white_check_mark:     |
| < 1.0  | :white_check_mark:（只有一个版本） |

不开玩笑。这是一个活的文档。只有一个版本。如果你发现了漏洞，它就在当前版本里。
我们会修复的。为一个 markdown 文件谈"受支持的版本"这个概念，几乎跟让 AI 不经
审查就写生产代码一样荒谬——而这也恰好是整个仓库想表达的核心观点。
